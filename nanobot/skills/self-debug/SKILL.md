# Skill 🩹 Self-Debug (异常中断与理性自愈)

## 🛠️ 处理逻辑 (Algorithm)

### 1. 保护现场 (Saving Context)
当你遇到任何工具报错（Exit code != 0）、文件缺失或逻辑执行结果与预期严重不符时，**严禁跳过错误强行推进**。
- **保存断点**：第一时间使用 `edit_file` 在 `HEARTBEAT.md` 的 `Active Missions` 顶部插入一条符合 Schema 的中断项。
- **格式规范 (必须严格遵守)**:
    `1. [ ] [Interrupt: Level N] 修补 [错误简述] ([当前项目名]) [Critic]`
    `    1.1. [ ] 故障溯源: 读取日志与源码，抽丝剥茧寻找根源。`
    `    1.2. [ ] 物理修复: 修改源码或配置，执行最小化改动。`
    `    1.3. [ ] 归位验证: 重新运行失败命令并执行物理断言。`
- **嵌套自愈 (Nested Interrupts)**：如果在执行本 Debug 技能的过程中再次报错，请将 Level 加 1，继续在顶部插入新的中断项。**禁止跳跃，必须先解决最深层（顶部）的那个 Bug。**

### 2. 理性溯源 (Rational Root-Cause Analysis)
禁止盲目尝试。你必须像侦探一样，理性地、一步一步抽丝剥茧：
- **查看物证**：优先读取 `exec("tail -n 100 /proc/1/fd/1")` 获取真实 Traceback。
- **环境审计**：使用 `ls -la` 亲自确认物理文件的存在、权限及其在 Docker 容器内的真实路径。
- **逻辑比对**：读取源码中相关函数的 `import` 关系与变量作用域，排查 `NameError` 或语义逻辑。
- **最小变量**：一次只修复一个可疑点，严禁大面积重写。

### 3. 步进修复与验证 (Step-by-Step Healing)
- **物理修复**：执行 `edit_file` 修改目标。
- **强制回归**：修改后，必须立即重新执行刚才失败的动作。
- **结果断言**：只有在工具返回了正确的数据后，才认为该 Level 的中断已解除。

### 4. 堆栈回溯 (Stack Popping)
- 修复成功后，修改 `HEARTBEAT.md` 勾选对应的子任务，并将对应的顶层中断项标记为 `[x]`（归档）。
- **接续现场**：重新读取看板中下一个 `[ ]` 状态的任务信息，恢复思维链继续执行。
